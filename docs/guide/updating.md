# Updating

## The launcher itself

The published image lives at `ghcr.io/wisp-hq/wisp:latest`. The Compose file generated by the installer references it.

```bash
docker compose pull
docker compose up -d
```

That's the whole update. Your `pb_data/` and `data/` directories are bind-mounted, so user accounts and per-user app data survive the swap.

::: tip Pin to a version
`:latest` is convenient but unpredictable. To pin:

```bash
# in .env
WISP_IMAGE=ghcr.io/wisp-hq/wisp:v0.5.0
```

Available tags are on the [GHCR package page](https://github.com/wisp-hq/wisp/pkgs/container/wisp).
:::

## App images

Each catalog entry is its own Docker image (e.g. `lscr.io/linuxserver/firefox:latest`). When you install an app, wisp pulls the image at install time and caches it on the host.

To refresh:

- **Admin UI** — the catalog tile shows an "Update" action that re-pulls the image (SSE-streamed progress, same as a fresh install).
- **API** — `POST /api/apps/<id>/update` triggers the same thing.

Already-running sessions keep their old image until stopped; the next launch picks up the new one.

## PocketBase / collection migrations

Migrations live in the launcher binary and run automatically on container start. When you `docker compose pull` a newer image, any new migrations run as the container boots — no manual step required.

If a migration fails on startup, the container logs the error and exits. Read [Troubleshooting → Container won't start](./troubleshooting#container-won-t-start).

## Rolling back

Worst-case rollback:

```bash
# Pin the previous version in .env
WISP_IMAGE=ghcr.io/wisp-hq/wisp:v0.4.0

docker compose up -d
```

::: warning DB migrations are forward-only
A version that introduces a destructive PocketBase migration won't downgrade cleanly — there's no "down" migration. Snapshot `pb_data/` before major version bumps if you want a safety net:

```bash
docker compose stop launcher
tar czf pb_data.backup.$(date +%F).tar.gz pb_data/
docker compose start launcher
```
:::

## Backing up

Two directories matter:

- `pb_data/` — user accounts, app records, sessions. Small.
- `data/` — per-user app data (browser profiles, save files, …). Can grow large depending on the apps.

A boring `tar` or `rsync` of those directories (with the launcher stopped, or with a brief read-lock) is enough. There's no schema-aware backup tool — PocketBase is a single SQLite file under `pb_data/`.
