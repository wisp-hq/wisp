# Architecture

wisp runs as a single Go process that embeds the React SPA, PocketBase, and a Docker-backed session manager. Browser traffic for streamed apps is reverse-proxied to per-user containers spawned on demand.

## Topology

```
                              ┌──────────────────────────────┐
   browser ──── /s/abc/* ────►│ wisp (one container)         │
                              │ ┌──────────────────────────┐ │
                              │ │ Go HTTP server (:8080)   │ │
                              │ │  ├─ /api/sessions  (Go)  │ │
                              │ │  ├─ /api/apps     (Go)   │ │
                              │ │  ├─ /api/collections (PB)│ │
                              │ │  ├─ /_/ admin UI    (PB) │ │
                              │ │  ├─ /s/<id>/* → proxy ───┼─┼─► spawned container
                              │ │  └─ / → SPA (embed)      │ │   (linuxserver/firefox,
                              │ └──────────────────────────┘ │    steam, …)
                              │ /var/run/docker.sock ◄───────┼── spawns via Docker SDK
                              └──────────────────────────────┘
```

## Spawned session containers

Each session container:

- joins the shared `launcher-net` Docker network so the launcher can reach it by IP
- is labelled `launcher.session=<sid>`, `launcher.user=<uid>`, `launcher.app=<slug>` so cleanup can find it
- gets per-user bind mounts (`/data/users/<uid>/...`) and read-only shared mounts (`/data/shared/games`, …) according to the app's `volumeConfig`
- receives `BASE_URL=/s/<sid>` so Selkies emits correctly-prefixed asset paths (the proxy also strips the prefix as a fallback)

## Cleanup

A goroutine reaps any session whose `lastActiveAt` is older than `IDLE_TIMEOUT` (default 30 min). It also removes orphan containers left over from previous launcher boots, found via the `launcher.session=…` label.

## Realtime data flow

The client side is built around **TanStack DB collections** that subscribe to PocketBase realtime channels. Anything visible in the UI — installed apps, session status, image-pull progress — comes from these collections rather than ad-hoc fetches. Mutations are kept as pure functions in [`client/src/mutations/`](https://github.com/wisp-hq/wisp/tree/main/client/src/mutations) and operate on the collection (`insert`, `delete`, …), letting the optimistic update layer handle UI feedback.

Image pulls are an exception: progress streams over an **SSE endpoint** (`/api/apps/:id/progress`) because PB realtime can't carry the byte-level granularity well. The SSE stream feeds a small in-memory store that the home-screen tile reads from.

## Streaming layer

The browser opens `/s/<sid>/` and the launcher's `httputil.ReverseProxy`:

1. Resolves the session record (cached in memory)
2. Strips the `/s/<sid>` prefix
3. Forwards HTTP and WebSocket traffic to `containerIp:3000`
4. Bumps an in-memory `lastActiveAt` buffer that the cleanup goroutine flushes every `CLEANUP_INTERVAL`

A second F5 on the same URL reuses the existing container — no respawn.

## Type sharing

The client used to import types from a `shared/` workspace generated by TypeScript. With the server in Go, the source of truth is the PocketBase collections themselves; the client's collection types live in [`client/src/lib/types.ts`](https://github.com/wisp-hq/wisp/blob/main/client/src/lib/types.ts) and are kept in sync by hand whenever a migration in `server/migrations/` changes a shape. This is intentionally low-tech — there are only a few collections and they change rarely.
